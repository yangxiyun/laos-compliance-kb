# SKILL: laos-compliance-kb-ingest

## 一句话描述
自动摄入法规文件到老挝合规知识库，生成结构化Wiki页面，更新Index和Log，推送GitHub。

## 触发条件
当用户说以下任何一种时触发本Skill：
- "请摄入"
- "摄入这个文件"
- "处理待摄入文件夹"
- "ingest"
- "更新知识库"
- "把这个加入知识库"

## 执行步骤（严格按顺序）

### Step 1：读取元文档

```
读取 D:\laos-wiki\META\SCHEMA.md
读取 D:\laos-wiki\META\Index.md
```

理解知识库的：
- 现有页面结构
- Wiki页面格式规范
- 文件命名规范
- 分类体系

### Step 2：扫描待摄入文件夹

```
扫描 D:\laos-wiki\待摄入\
```

列出所有文件，支持格式：
- `.pdf` — 法规文件、报告
- `.md` — Markdown笔记、整理好的文字
- `.txt` — 纯文本
- `.docx` / `.doc` — Word文件
- `.html` / `.htm` — 网页存档

如果文件夹为空，告知用户并结束。

### Step 3：逐一处理每个文件

对每个文件执行：

**3a. 读取文件内容**
- PDF：提取全部文本内容
- 其他格式：直接读取

**3b. 分析内容，判断分类**

按以下优先级判断：
- 包含"增值税/VAT"关键词 → 税务/增值税
- 包含"企业所得税/利润税/CIT" → 税务/企业所得税
- 包含"个人所得税/PIT/工资税" → 税务/个人所得税
- 包含"预提税/FWHT/WHT" → 税务/预提税
- 包含"转让定价/关联交易" → 税务/转让定价
- 包含"会计/财务报表/审计" → 会计/对应子类
- 包含"注册/营业执照/工商" → 工商/对应子类
- 包含"劳动/员工/薪酬/社保" → 人力资源/对应子类
- 包含客户案例/咨询方案内容 → 业务经验库/对应子类
- 无法判断 → 询问用户确认分类

**3c. 决定生成页面数量**

| 文件类型 | 生成页面数 |
|---|---|
| 单一税种法规（如增值税法） | 2-3个（概览+税率+申报） |
| 综合性法规（如所得税法） | 3-5个（按章节拆分） |
| 政策文件、通知 | 1个 |
| 业务案例、咨询报告 | 1个（脱敏处理） |
| 操作指南、实务手册 | 2-3个 |

**3d. 生成Wiki页面**

严格按以下格式生成每个页面：

```markdown
---
分类：[税务/会计/工商/人力资源/业务经验]
子类：[具体子分类]
法规依据：[相关法规全称及条款]
创建日期：[YYYY-MM-DD]
最后更新：[YYYY-MM-DD]
相关页面：[[页面A]] [[页面B]]
---

# 页面标题

## 核心要点
（3-5条最重要的结论，直接可用）

## 详细内容
（结构化正文，分章节展开）

## 实务注意事项
（容易出错的地方，ZTL实践经验）

## 关键条款摘录
（法规原文关键段落，注明出处和条款号）

## 相关页面
（链接到知识库内相关条目）
```

**3e. 写入文件**

```
写入路径：D:\laos-wiki\[分类]\[子类]\[文件名].md
文件命名：[子类]-[主题].md
示例：增值税-申报流程.md
```

**3f. 更新 Index.md**

在对应分类下追加：

```markdown
| [[页面标题]] | 一句话摘要 | 创建日期 |
```

同时更新顶部的：
```
最后更新：YYYY-MM-DD
Wiki页面总数：N
```

**3g. 更新 Log.md**

在顶部插入新记录：

```markdown
## YYYY-MM-DD

**操作类型：** 摄入
**原始资料：** 文件名
**生成页面：** 页面1、页面2、页面3
**操作人：** Xiyun
**备注：** 简要说明
```

**3h. 移动原始文件**

```
移动：D:\laos-wiki\待摄入\文件名
到：  D:\laos-wiki\处理完成\文件名
```

### Step 4：业务经验类特殊处理

如果文件属于业务经验库，执行脱敏：
- 客户名称 → 用行业代称（橡胶客户A、建筑客户B）
- 具体金额 → 保留数量级（约XXX万基普）
- 联系方式 → 完全删除
- 内部邮件内容 → 完全删除
- 保留：问题类型、处理思路、结论、法规依据

### Step 5：Git推送

```bash
cd D:\laos-wiki
git add .
git commit -m "摄入: [文件名列表] - YYYY-MM-DD"
git push origin main
```

### Step 6：输出摄入报告

每次摄入完成后，输出以下摘要：

```
✅ 摄入完成报告
================
处理文件：N个
生成Wiki页面：N个
  - [分类/子类/文件名.md]
  - ...
更新Index：已添加N条
推送GitHub：成功/失败
耗时：约N分钟

⚠️ 注意事项（如有）：
- [需要用户确认的事项]
- [知识库中发现的关联页面需要更新]
```

## 查询模式

当用户问问题（非摄入请求）时：

### Step 1：读取Index
```
读取 D:\laos-wiki\META\Index.md
```

### Step 2：定位相关页面
根据问题关键词，从Index中找到最相关的1-3个页面。

### Step 3：读取页面内容
```
读取 D:\laos-wiki\[分类]\[子类]\[文件名].md
```

### Step 4：综合作答
- 注明引用来源（页面名称）
- 如知识库无相关内容，明确说明并建议摄入哪类资料

## 注意事项

1. **必须先读SCHEMA.md**，每次执行前都要读，不能依赖记忆
2. **Index.md 是唯一的页面目录**，每次摄入后必须更新
3. **Log.md 是完整操作历史**，每次操作都要记录
4. **业务经验类必须脱敏**，不得出现客户真实名称
5. **文件命名用中文**，与现有文件夹风格保持一致
6. **双向链接**：生成页面时检查与已有页面的关联，主动加入 `[[链接]]`
7. **Git推送失败不影响结果**，提醒用户手动推送即可

---

## 常见问题处理

| 情况 | 处理方式 |
|---|---|
| 文件内容为空或无法读取 | 告知用户，跳过此文件 |
| 无法判断分类 | 询问用户确认后继续 |
| 文件内容超大（>100页） | 分段处理，生成多个页面 |
| 已有同名Wiki页面 | 询问是否覆盖或合并 |
| Git推送失败 | 提醒用户手动推送，不影响本地文件 |
| 业务经验类脱敏不确定 | 列出不确定项，询问用户确认 |
